## Motivation

The server has an implicit runner state machine — \`healthy → stale → healthy\` (or \`→ drained\`) — but only emits an event on entering \`stale\` (\`runner.heartbeat_missed\`, pool.rs:186-192). The exit from \`stale\` back to \`healthy\` is a silent \`already_missed.retain(...)\` mutation on the server (pool.rs:199-215).

This asymmetry directly caused the bug in sKFs — the herder (a pure event-stream projection) had no way to re-learn about a runner that recovered from a transient stall. Both state boundaries need events; otherwise any projection tracking runner liveness collapses stale → dead and strands recovered runners.

## Scope

1. When the server's staleness sweep clears a runner from \`already_missed\` (transition stale → healthy), emit a \`runner.recovered\` event with the runner_id and \`last_seen\`.
2. Herder handler for \`runner.recovered\`:
   - If runner is missing from local map (shouldn't be, after the current sKFs fix, but defensive), re-insert as idle.
   - If runner is in map and currently \`idle = false\` with a \`current_execution/step\` that no longer matches the server's view (we may not be able to detect this cleanly yet), log a warning.
3. Invariant: for every state a runner can occupy, both the entry and exit of that state are events. No silent mutations.

## Out of scope

- Splitting runner liveness into an explicit \`healthy/stale/dead\` enum on the projection. The new event gives us the signal; we can do the refactor later if it pays off.
