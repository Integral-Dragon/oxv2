ox

GitHub for agents.

Event-sourced workflow engine for AI agents. Multi-agent teams in isolated sandboxes, coordinated by a human.

the pitch

Issues track work. Pull requests are workflow executions. Runners are fungible VMs. Events drive everything. You play the repo owner — setting direction, reviewing escalations, controlling budget — while agents handle contribution.

Personas define who an agent is. Skills define what they can do. Workflows define what runs, when, and how. The engine is generic. Complex multi-agent orchestrations are configuration, not code.

five rings

  1. engine — event-sourced workflow execution with isolated runners. runs on your laptop.
  2. skills — capability packages (tools, scripts, instructions) composable at every level.
  3. ecosystem — a registry for sharing skills, personas, and workflows.
  4. platform — hosted infra: cloud runners, GitHub integration, web dashboard.
  5. self‑improvement — retro workflows that review history and update memory. agents get better over time.

three components

ox-server

the hub. owns shared state, exposes HTTP + SSE, the single writer to main. passive — only responds to requests.

ox-herder

the active loop. reads from ox-server, counts things, fires triggers. not AI. dumb on purpose.

ox-runner

the executor. stateless and fungible. runs anywhere — seguro VM, cloud, k8s, bare metal. registers and waits.

principles

  • event sourced. all state is a projection of an append-only log. history is always recoverable.
  • single writer to main. nothing writes to main except ox-server's merge. the merge is the serialisation point.
  • fungible executors. runners have no identity. any runner can run any step. pool size is the WIP limit.
  • branch discipline for everyone. agent or human. all work happens on branches and merges when done.
  • git log as event source. cx state changes are derived from git log. no separate event files.
  • two‑phase completion. a step isn't done until the runner confirms it. the herder never races in-flight work.
  • personas are primary. workflows name personas, not runtimes. swap a persona, swap everything about how the step runs.
  • orchestration is configuration. personas, skills, workflows are files. the infrastructure is generic.
  • skills are portable. files on disk, executables on PATH. no protocol dependency. any agent can use them.
  • the herder is dumb. no AI in the infrastructure layer. intelligence lives in agents and personas.
